使用Github和Hexo,Travis CI搭建自动化构建的独立博客站点

发表于 2019-05-29 | 更新于 2019-06-03 | 分类于 实用技巧 | 阅读次数：
本文字数： 5.4k | 阅读时长 ≈ 5 分钟

目标

我们先来看看本文希望达成怎样的目标:

  1. 使用 Hexo 静态页面生成的博客系统搭建个人博客站点。
  2. 使用 Github 的私有仓库作为博客站点的草稿箱，草稿箱内的所有数据不提供公开访问。这样我们可以使用 Github 的 Iuuse 管理工具进行写作的计划安排以及参考资料的收集，同时也可以使用 Pull Request 进行实际的写作管理。
  3. 使用 Github Pages 的公开仓库作为博客站点的对外服务站点。
  4. 整个博客发布流程完全使用免费工具(Azure Devops、Travis CI)全自动化完成。

完成后的整个构建发布过程如下：

准备工作

  1. 本地环境
    • NodeJS
    • Git
  2. Github 账号一枚，同时创建一个个人授权访问码(PAT:Personal access tokens)，该授权码需要具备 repo:public_repo 访问权限且只要这个权限。
  3. Azure Devops 账号一枚。

建立 Github Repository

我们需要建立两个仓库：

  • 一个用于储存、管理博客文章草稿状态的私有仓库，不对外公开访问，在这个仓库中完成博客的日常写作，一旦完成一篇文章，我们就将该文章推送到该私有库的 master 分支，后续所有构建和发布环节均为自动完成。同样的，我们对博客进行的配置和设置也应该在这个私有库中进行，配置完成后由自动化构建过程发布到博客站点。
  • 一个用于对外提供访问的公开仓库，使用 Github Pages 服务，可以很轻松的实现全静态站点，同时也可以绑定自定义域名等。根据 Github Pages 的使用规范要求，该仓库的命名必须是 <github用户名>.github.io，如兜小神的 Github 用户名是 kelely，这个仓库的名称就是 kelely.github.io，只有按照这个要求命名，才可以绑定到顶级域名上，否则就只能通过域名+子目录的形式访问。具体说明这里不展开说了，官方介绍中有详细的说明。当我们的私有库中 master 产生了新的提交，最终会将最新的构建结果发布到该仓库的 master 分支，实现对外的访问。

建立私有草稿仓库

先在Github上建立一个私有仓库，仓库名字可以根据个人的喜好来定，为了便于介绍，本文中以仓库名为: your-blog-private-repo 举例说明, 这个名字会出现在后续中的多个配置，所以请一定要确保后续配置中的名字和这个名字匹配。

建立公开博客站点仓库

根据 Github Pages 的使用规范要求，创建一个名为 <github用户名>.github.io 的公开仓库，创建时顺便进行初始化，直接生成 master 分支，也可以手动初始化，总之目的就是要产生一个 master 分支，如果是个空仓库，在后续同步到该仓库的过程是会有问题的。

配置 CI 流程

由于我们采用了私有仓库进行博客文章的写作和管理，而目前市面上支持对 Github 私有仓库进行自动化构建的 CI 工具几乎没有，得益于微软爸爸收购 Github 后的深度融合，微软旗下的 Azure Devops 对 Github 的私有仓库支持自动化构建了，所以我们在这里选用 Azure Devops 来进行自动化构建，在本人实践的过程中，发现一个小问题，那就是 Azure Devops Pipelines 在构建过程中无法获取代码仓库中源地址是 Github 的 Sub-Modules，而 Hexo 使用主题的最佳实践方式就是使用 Sub-Modules，所以这里整个 CI 流程不得不采取一个妥协的策略：使用 Azure 将私有库中的 master 分支同步到公开库中的临时镜像分支 mirror，再使用 Travis CI 对 mirror 分支进行构建，构建的最终工件由 Travis CI 推送到 公开库中的 master 分支，同时由 Travis CI 将公开库中的临时镜像分支删除。如果你不使用 Sub-Modules 的方式安装部署 Hexo 主题，也可以只用 Azure Devops Pipelines 完成整个构建过程。

然后登录到 Azure Devops，建立一个新项目，名字随意，自己喜欢就好，在这个项目内进行 Pipelines 的配置。
添加一个变量分组: Pipelines > Library > + Variable group
分组名可以自由指定，但一定要确认和 azure-pipelines.yml 文件中的变量分组名一致，否则构建过程会找不到对应的变量，分组内建立一个变量名为 GITHUB_TOKEN 的环境变量，变量值是准备阶段创建好的 Github PAT，构建过程中需要通过这个授权码在公开仓库中建立镜像分支。

创建一个构建过程： Pipelines > Builds > New Pipeline
选择 Github(YAML)，然后登录Github账号进行授权，选择前面已经建立好的私有仓库：your-blog-private-repo，
Azure 会自动找到根目录下的 azure-pipelines.yml，进行构建，如果没找到这个文件，请确认这个文件是否已经推送到私有仓库的根目录。
现在可以直接运行一次同步构建了，验证配置是否正确，构建完成后，应该可以在 <github用户名>.github.io 这个公开仓库中看到创建的 mirror 分支，并且是私有库 master 分支的镜像。

Azure 配置完成。
